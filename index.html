Froggie Hop

by: Richardson Liou

This game was made for my level 2 project at The League of Amazing Programmers.
Click here to download and play!

1. Why I Made This Game

I decided to make this game because it is similar to a game I used to play a lot. I also saw different versions of this game but with different characters and backgrounds. I wanted to customize it myself as well so I decided to make this game. This game can also be played for a long time. Another game I wanted to make was a game like temple run, but making those graphics are a lot harder and they probably wouldn't have looked good.

2. What I Learned

I learned from user testing that my game could be really easy and repetitive. My classmate told me that I should make the game more difficult once you reach a certain score so that the game didn't seem so easy, and to make platforms random sizes.

3. Architecture of My Game

The game panel class created what was on the window and where objects would be. The game panel had 2 states: the game state, and the menu state. The runner class would display which state the game panel was in on the window. The platform and player classes were extended from the game object class, which had the basic position and size variables for each object. The object manager class was what updated the player and platforms, spawned in and drew platforms, checked if the player had died, and made the screen transition up if the player reached a certain height. It also checked if the player collided with a platform and made it stay on the platform, unless the player was moving upward from below that platform.

4. Technical Implementation

When the player is jumping from platform to platform, the platform first needs to check if the player is moving upwards or downwards when they collide. If the player is moving upwards, the platform lets the player jump through itself. If it is moving downwards when landing on the platform, they will collide and the player will stop and land. Once the player reaches a certain height on the screen, the screen will scroll upwards and spawn in more platforms to continue the game. To accomplish this, I had to first find how to switch screens and clear the old platforms to spawn new platforms. Then I changed it to smoothly transition to where the player was moving up to, instead of just snapping to the next screen.

5. Most Challenging Part

One of the most challenging parts of the project was finding the perfect values to make platforms spawn without being to close to each other, but still close enough so that jumping from each platform wasn't impossible. To do this, I had to calculate how high the player could jump on each jump, and then changing the function that spawned platforms to spawn them at a certain minimum and maximum value from the previous platform. My teacher also helped me create a smooth transition to move to the next set of platforms for the player.

6. Future Fixes and Updates

I would like to fix the images of the smaller platforms to be a different picture from bigger platforms, so that they don't look so squished. I also want to make more obstacles for the player in the future, like moving platforms and platforms that break after a certain amount of time after landing on them.
